# Quick Task 260420-a01 Summary

**Android SOS alerts now use a permission-aware escalation path that prefers local full-screen alerts, native DND-bypass channel updates, and honest fallback copy without changing iOS behavior.**

## Accomplishments

- Added a native Android SOS bridge so Flutter can check Notification Policy Access, check full-screen intent capability, open the relevant settings screens, and recreate the SOS notification channel with `setBypassDnd(true)` when Android allows it.
- Shifted Android SOS delivery toward a stronger local-rendered path by sending data-first SOS payloads for known Android installations and routing the background FCM isolate through `SosNotificationService` before generic notification handling.
- Preserved graceful fallback behavior by keeping legacy system-rendered SOS payloads for unknown-platform tokens, leaving iOS sound behavior unchanged, and updating Android-facing copy to describe the feature as best-effort rather than guaranteed mute bypass.

## Android Files Touched

- `android/app/src/main/AndroidManifest.xml` - Added `ACCESS_NOTIFICATION_POLICY`, `USE_FULL_SCREEN_INTENT`, lock-screen activity flags, and the local-notification action receiver.
- `android/app/src/main/kotlin/com/example/kid_manager/MainActivity.kt` - Added the `sos_alerts` method channel, SOS channel recreation logic, DND/full-screen settings deep links, and stronger alarm-style channel configuration.

## Flutter Files Touched

- `lib/main.dart` - Routed Android background SOS messages through the local escalation path when the payload is marked for local rendering.
- `lib/services/notifications/android_sos_alert_bridge.dart` - Added the Flutter bridge for native SOS alert capabilities and settings entrypoints.
- `lib/services/notifications/sos_notification_service.dart` - Added native-backed channel sync, background-safe initialization, full-screen local SOS display, stronger vibration/category settings, and truthful iOS Critical Alerts comments.
- `lib/services/permission_service.dart` - Added Android SOS-specific permission/status helpers and a best-effort settings flow.
- `lib/features/permissions/notification_permission_screen.dart` - Allowed custom Android SOS settings copy.
- `lib/features/permissions/permission_onboarding_flow.dart` - Added Android SOS helper text and routed settings to the new SOS-specific permission flow.
- `lib/features/permissions/permission_step_scaffold.dart` - Added helper-text support for permission guidance copy.
- `lib/features/permissions/sos_prelaunch_gate.dart` - Updated the SOS recommendation/settings CTA to explain Android limits honestly.

## Backend Files Touched

- `functions/src/services/sosPush.ts` - Split SOS payload composition by platform so Android can receive data-only urgent alerts while iOS and unknown-platform fallbacks keep the existing notification-based path.
- `functions/test/sos_push_payloads.test.mjs` - Added focused payload-shape coverage for Android, iOS, and unknown-platform SOS fanout.

## User Setup Required

- Android users who want the strongest SOS behavior must enable normal notification permission first.
- On Android, users should also grant Notification Policy Access if they want the SOS channel to request DND bypass.
- On Android 14+, users may also need to allow full-screen intents for the app.

## Remaining Platform Limits

- Third-party Android apps still cannot guarantee sound in every vendor-specific silent-mode configuration.
- DND bypass only applies when Android grants Notification Policy Access and the device honors that channel setting.
- iOS mute/DND bypass remains unsupported here because Apple Critical Alerts entitlement is intentionally out of scope.
